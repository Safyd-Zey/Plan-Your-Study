# Assignment 2: Automation Strategy and Quality Metrics

## 1. Введение

Этот документ описывает автоматизацию тестирования для проекта "Plan Your Study" в рамках задания 2. Целью является применение стратегии автоматизации из задания 1, определение качественных ворот и сбор метрик, которые будут использованы в исследовательской работе.

## 2. Система и область автоматизации

Тип системы: веб-приложение с полнофункциональным frontend на React + TypeScript и backend на FastAPI.

Основные зоны автоматизации:
- API аутентификация и авторизация
- CRUD курсов
- Управление заданиями и статусы
- Подзадачи и изменение состояния задач
- Прогресс и метрики выполнения
- Расписание и календарь
- Основные пользовательские сценарии через UI (регистрация, создание курса, создание задания)

## 3. Выбор инструментов и мотивация

### 3.1 API / backend
- **Pytest** — стандартный инструмент для Python, удобен для интеграционных тестов FastAPI.
- **pytest-cov** — сбор покрытия кода для определения процента протестированных модулей.

### 3.2 Web UI / frontend
- **Playwright** — современный фреймворк для end-to-end тестирования браузера, подходит для React/Vite приложений.
- **Vite** / веб-сервер — тесты запускаются против живого фронтенд-приложения.

### 3.3 CI/CD
- **GitHub Actions** — автоматический запуск тестов при push, pull request и по расписанию.
- Файл: `.github/workflows/test.yml`

### 3.4 Почему этот стек
- Система является web-приложением, поэтому сочетание API тестов и E2E покрывает критические высокорисковые сценарии.
- Используется уже существующий стек проекта и готовая CI-конфигурация.
- Позволяет документировать результаты и метрики для дальнейшего исследования.

## 4. Реализованные автоматизированные проверки

### 4.1 Бэкенд тесты

Файлы тестов:
- `tests/conftest.py` – фикстуры, изоляция базы данных, конфигурация клиента
- `tests/integration/test_api_auth.py` – регистрация, логин, получение текущего пользователя, изоляция данных
- `tests/integration/test_api_course_assignment_subtask_progress.py` – курс, задания, подзадачи, прогресс
- `tests/integration/test_api_schedule.py` – расчёт расписания, генерация календаря
- `tests/unit/test_auth.py` – модульные проверки аутентификации

Ключевые автоматизированные сценарии:
- Регистрация пользователя, повторная регистрация с тем же email
- Вход пользователя, ошибка при неверном пароле
- Создание курса, получение списка, изоляция между пользователями
- Жизненный цикл задания: создание, чтение, обновление, фильтры по курсу и дате, завершение, удаление
- Подзадачи: создание, изменение, завершение, удаление
- Переход статуса задания при изменении подзадач
- Прогресс: общее количество, завершённые, в работе, процент завершения
- Расписание: календарь по месяцу, отложенные и предстоящие задания

## 5. Интеграция CI/CD

Файл: `.github/workflows/test.yml`

### 5.1 Структура pipeline

1. **backend-tests**
   - Запуск на Ubuntu
   - Матрица Python: 3.8, 3.9, 3.10
   - Установка зависимостей
   - Запуск `pytest ../tests/ -v --cov=backend --cov-report=xml --cov-report=term`
   - Загрузка покрытия в Codecov

2. **frontend-e2e-tests**
   - Выполняется после успешного `backend-tests`
   - Установка Node.js 20
   - Установка frontend-зависимостей и Playwright
   - Запуск backend и frontend сервисов
   - Прогон Playwright тестов через `npx playwright test --config=playwright.config.ts`

3. **code-health**
   - flake8 базовая проверка на ошибки синтаксиса и качества
   - black --check для контроля форматирования

### 5.2 Точки проверки в CI

- Тесты должны завершиться успешно — любой провал останавливает ветку.
- Код покрытия должен генерироваться и отправляться в Codecov.
- Форматирование и базовая статическая проверка должны проходить.

## 6. Quality Gates (Качественные ворота)

### 6.1 Определённые критерии

- **Прохождение автоматических тестов**: все backend-интеграционные и unit тесты должны быть зелёными.
- **Порог покрытия**: целевой порог 80%+ для backend. Текущее фактическое значение 86%.
- **Критические дефекты**: нулевое количество критических ошибок в CI-проходе.
- **Время выполнения**: базовый backend-пакет должен укладываться в менее 30 секунд, E2E — менее 120 секунд в CI.
- **Статус сборки**: любые `pytest`, `Playwright` или `code-health` ошибки завершают сборку как failed.

### 6.2 Условия отказа

CI считается проваленным если:
- хотя бы один тест завершился с ошибкой
- покрытие снизилось ниже порога
- линтер/форматирование обнаружили ошибки
- фреймворк E2E не смог запустить приложение
